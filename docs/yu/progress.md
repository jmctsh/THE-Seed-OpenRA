## [2026-04-01 04:41] DONE — Root-cause investigation of Adjutant LLM hang in `main.py`
Traced the live path `RuntimeBridge.on_command_submit -> Adjutant.handle_player_input -> _classify_input/_handle_query -> llm.chat` and wrote the report to `docs/wang/adjutant_hang_investigation.md`. The root cause is architectural, not provider-specific: `ApplicationRuntime.start()` launches `GameLoop` on the same asyncio event loop that serves WS and awaits Adjutant/Qwen coroutines, but `GameLoop._tick()` directly calls synchronous `WorldModel.refresh()` and synchronous job ticks. `WorldModel.refresh()` in turn fans out to blocking `GameAPI` socket calls (actor queries, economy, map, and five production queues), and `GameAPI._send_request()` itself uses blocking sockets with 10s timeouts and retries. That means a slow game poll can starve the event loop long enough that even Adjutant's existing `asyncio.wait_for()` timeouts do not fire, making the LLM call look hung. I verified the diagnosis three ways: line-by-line code tracing; a minimal repro where a fake async LLM returns in ~0.10s without `GameLoop` but stalls when `GameLoop` uses `time.sleep(0.3)` in `world_model.refresh()`; and an `ApplicationRuntime`-shaped repro where `FastWorldSource` returns in ~0.10s but `BlockingWorldSource` gets stuck in repeated over-budget ticks until an external process alarm interrupts it. The recommended fix is to move GameLoop/game I/O onto a dedicated worker thread (or otherwise isolate blocking game polling from the main asyncio loop), with explicit provider HTTP timeouts as secondary hardening rather than the primary fix.
